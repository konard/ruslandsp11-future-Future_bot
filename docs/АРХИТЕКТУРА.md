# Структура и алгоритм работы Future_bot

Документ описывает устройство программы и алгоритм её работы: из каких модулей
состоит бот, как обрабатываются команды и как выполняется поиск постов ВКонтакте.

## Назначение

Бот следит за чатом ВКонтакте и по команде (или по расписанию) собирает свежие
посты из групп-источников, отбирает их по ключевым словам и присылает в чат
пронумерованный список ссылок вместе с кратким отчётом.

## Модули

| Файл | Ответственность |
| --- | --- |
| `future_bot/__main__.py` | Точка входа CLI: разбор аргументов, сборка сервиса, запуск нужного режима. |
| `future_bot/config.py` | Загрузка настроек из `.env`, чтение файлов групп и команд, валидация значений. |
| `future_bot/logic.py` | Чистая логика без ввода-вывода: разбор команд, разбор запросов, фильтрация постов, работа со ссылками. |
| `future_bot/service.py` | Оркестровка: сбор постов, применение запросов, отправка отчётов, обработка команд чата, фоновый поиск. |
| `future_bot/storage.py` | SQLite-хранилище: посты ФФ, новые посты, метаданные, удаление устаревших записей. |
| `future_bot/scheduler.py` | Циклы: ежедневный запуск по времени и постоянный опрос чата. |
| `future_bot/vk_client.py` | HTTP-клиент VK API (чтение стен, списки лайков, отправка и редактирование сообщений, настройка TLS). |

## Модель поискового запроса

Запрос представлен как **ИЛИ групп из И-слов** — тип
`tuple[tuple[str, ...], ...]`.

- Запятая — логическое **ИЛИ** между группами.
- Знак `+` — логическое **И** внутри группы (все слова должны присутствовать в
  одном посте, необязательно рядом).

Пример: `(ИИ+прогноз+технологии, антилопы+экология)` превращается в
`(("ИИ", "прогноз", "технологии"), ("антилопы", "экология"))`. Пост проходит
фильтр, если совпала хотя бы одна группа целиком.

Разбор выполняет `parse_query_groups`, а команду целиком — `parse_search_command`
по шаблону `/поиск по (…) интервал Nд`.

## Обособленное совпадение слов

Слово в посте должно быть **отдельным словом**, а не частью другого слова.
`_compile_term` компилирует терм в регулярное выражение
`(?<!\w)<слово>(?!\w)` с `re.IGNORECASE`. Так как в Python символы класса `\w`
включают кириллицу, это правило:

- не даёт `Маск` совпасть внутри `маскировка`;
- требует, чтобы `ИИ` было обособленным словом, а не серединой/концом другого;
- одновременно распознаёт хэштеги (`#ИИ`), потому что перед словом стоит `#` —
  небуквенный символ, то есть граница слова.

## Алгоритм одного поиска (`_run_search`)

1. Определяется список команд `_resolve_queries`:
   - если группы заданы явно (команда из чата) — это одна команда;
   - если запрос пустой (`/поиск по () …`) или это запуск по таймеру — команды
     берутся из файла `Список слов и хэштегов.txt`: **каждая строка — отдельная
     команда с отдельным отчётом**.
2. Инкрементально дозагружаются посты группы «Формула Футурологии» начиная с даты
   последнего сохранённого поста и сохраняются в таблицу `База постов ФФ`.
3. Устаревшие записи `База новых постов` удаляются (`_prune_old_posts`, см.
   «Хранение и лимиты»). `База постов ФФ` при этом не трогается.
4. Опрашиваются группы-источники за интервал `now - interval_days`. Прогресс
   поиска отражается редактированием сообщения не чаще раза в минуту с оценкой
   времени окончания. Ошибки отдельных групп логируются, поиск продолжается.
5. Результаты опроса источников дедуплицируются. Повторный запуск в течение 12
   часов при неизменном списке групп переиспользует последний опрос.
6. Для **каждой команды** отдельно:
   - посты фильтруются `filter_posts_by_groups` по её группам;
   - удаляются посты, ссылки на которые уже есть у постов ФФ (`remove_posts_linked_from_ff`);
   - удаляются посты, уже отмеченные лайком в ответе `wall.get`
     (`remove_liked_posts`, работает только с токеном пользователя);
   - удаляются посты, уже записанные в `База новых постов` (`remove_known_posts`),
     — так один и тот же пост не выдаётся дважды по разным строкам файла команд
     или в следующем поиске;
   - у оставшихся ссылок запрашивается `likes.getList`, и посты, лайкнутые
     админами чата, исключаются (`_remove_posts_liked_by_admins`) — лайк служит
     отметкой «уже просмотрено». Результаты кешируются на время поиска, а ошибка
     запроса лайков не убирает пост из отчёта;
   - формируется отчёт и пронумерованный список ссылок.
7. Все итоговые посты дописываются в `База новых постов` (`add_new_posts`), каждый отчёт
   отправляется в чат отдельным сообщением (первый — правкой сообщения о
   прогрессе), длинные списки разбиваются на части. Отчёт с заголовком (какие
   слова искались и по какому запросу выдан результат) формируется как для
   команд из чата, так и для планового запуска по таймеру (`run_forever`
   вызывает `run_once(include_summary=True)`).

## Обработка команд чата

- `poll_chat_once` читает последние сообщения, сортирует по порядковому номеру и
  обрабатывает только новые (по метаданным `last_processed_message_sequence`).
- **Игнорирование уже отвеченных команд** (`_command_already_answered`): если
  после команды в чате есть сообщение от имени группы (`from_id < 0`), значит
  команду уже обработал другой экземпляр бота, и она пропускается. Это исключает
  дублирование при развёртывании программы у нескольких пользователей.
- **Надёжная остановка поиска.** В режиме опроса чата команды поиска запускаются
  в фоновом потоке (`ThreadPoolExecutor`), поэтому поток опроса не блокируется и
  сразу распознаёт `/стоп поиск`. Команды управления обрабатываются немедленно и
  синхронно. Флаг остановки проверяется перед опросом каждой следующей группы.
- Команды доступны только пользователям из `FFBOT_ALLOWED_USER_IDS`.
- `parse_control_command` распознаёт варианты `/стоп`, `/стоп поиск` и т.п.
  (остановка поиска) и `/стоп программа`, `/выход` и т.п. (остановка программы),
  устойчив к регистру и завершающей пунктуации.

## Хранение и лимиты памяти

- `База постов ФФ` — накапливаемая таблица для дедупликации; дозагружается
  инкрементально (со смещением, начиная с последнего поста) и только
  дополняется постами группы ФФ. Она не считается устаревшей и не подчиняется
  ретенции — записи из неё никогда не удаляются по возрасту.
- `База новых постов` — накопленные выданные посты; каждый успешный поиск
  дописывает в неё свои результаты (`add_new_posts`). Именно она служит
  памятью о том, какие посты уже отправлялись в чат, поэтому размер таблицы
  ограничивает ретенция, а не полная перезапись.
- **Ретенция**: `FFBOT_POST_RETENTION_DAYS` (по умолчанию 10) задаёт, за сколько
  дней хранить посты в `База новых постов`. `delete_posts_older_than` удаляет
  из неё записи с датой раньше `now - N дней`, поэтому посты возрастом N дней
  сохраняются, а более старые (например, 11-дневной давности при N = 10)
  стираются. Это ограничивает объём базы новых постов, оставшихся от прежних
  (не завершившихся успешно) поисков, и не влияет на `База постов ФФ`.

## Режимы запуска

- `--once` — один цикл поиска.
- `--daemon` — ежедневный запуск в `FFBOT_SCHEDULE_TIME`.
- `--poll-chat` — постоянный опрос чата на команды.
- `--daemon --poll-chat` — оба режима одновременно (`run_forever_with_chat_polling`).

## Токены и доступ к VK API

- `VK_SERVICE_TOKEN` — сервисный токен приложения; им читаются стены групп
  (`wall.get`) и списки лайков (`likes.getList`). Метод `likes.isLiked`
  сервисному токену недоступен, поэтому принадлежность лайка определяется по
  списку лайкнувших пользователей.
- `VK_USER_TOKEN` — прежний токен пользователя, используется как запасной
  вариант, если сервисный токен не задан.
- `VK_GROUP_TOKEN` / `VK_MESSAGE_TOKEN` — работа с чатом и отправка сообщений.

## TLS и устойчивость сети

- `build_https_opener` создаёт `SSLContext` с явно выбранным файлом корневых
  сертификатов: `FFBOT_VK_CA_BUNDLE`, затем `SSL_CERT_FILE` / `REQUESTS_CA_BUNDLE`,
  затем `certifi` (если установлен), иначе системное хранилище. Это устраняет
  ошибку `[SSL: CERTIFICATE_VERIFY_FAILED] unable to get local issuer certificate`,
  из-за которой падал опрос чата.
- `FFBOT_VK_SSL_VERIFY=0` полностью отключает проверку сертификатов — только для
  диагностики.
- Временные сетевые ошибки (`URLError`, кроме `HTTPError`) повторяются
  `NETWORK_RETRY_ATTEMPTS` раз с паузой; ошибки VK API повторно не отправляются,
  кроме отдельной обработки `Flood control` (код 9).
